# 深度問答練習 — NGO 管理系統

> 這份文件模擬面試官追問的情境，每題都有「標準答案」和「如果被繼續追問」兩層。
> 目標是讓你能**自信流暢地說出每個決策背後的 why**，而不是背程式碼。
>

## 一、系統架構

---

**Q1：你這個系統架構是怎麼設計的？**

> 三層架構：ASP.NET Core 9 WebAPI 做後端統一入口，前端有兩套——React 18 管理後台和 ASP.NET Core MVC 用戶前台，資料庫是 SQL Server。兩個前端都共用同一個 API，商業邏輯不重複。

**追問：為什麼要兩個前端，不能一個 React 搞定？**

> 這是刻意的設計決策。員工後台需要大量互動操作，React SPA 的體驗好很多；用戶前台是給一般民眾和弱勢個案用的，考慮首次造訪的 SEO、載入速度，以及讓熟悉傳統網站的用戶更好上手，MVC SSR 更合適。不是技術能力不夠，而是根據使用者特性選不同方案。

---

**Q2：你的後端分層怎麼做的？**

> Controller → Service → Repository → DbContext。Controller 只做請求驗證和回應格式；Service 放商業邏輯；Repository 封裝資料存取。目前 Repository Pattern 只在 Case 模組完整實作，其他模組 Controller 直接用 DbContext。

**追問：那為什麼不統一全部都用 Repository？**

> 因為過度抽象也是問題。Case 模組有複雜的多表查詢、狀態機邏輯，Repository 的價值最大；其他模組如 Supply、Activity 的查詢相對單純，加 Repository 只是增加樣板程式碼。我把「需要複雜查詢」和「需要被單元測試」作為引入 Repository 的門檻，而不是為了架構整齊而強制套用。

---

**Q3：API 回應格式怎麼處理的？**

> 全站統一用 `ApiResponse<T>`：`{ success: bool, message: string, data: T }`。成功回傳 `success: true` 加上 `data`；失敗回傳 `success: false` 加上 `message`。React 前端的所有 service 都統一從 `response.data` 解包，不需要每個元件自己處理不同格式。

**追問：Global Exception Handler 呢？**

> API 和 MVC 都有。API 端在 `Program.cs` 用 `UseExceptionHandler` 攔截所有未處理例外，回傳標準化 `{ success: false, message, traceId }` JSON，並用 `ILogger` 記錄完整 stack trace。MVC 端用 `UseStatusCodePagesWithReExecute` 處理 4xx/5xx，統一導向 `HomeController.Error`，顯示對應的錯誤頁面，不讓 stack trace 外洩給使用者。

---

## 二、認證與授權

---

**Q4：你怎麼做認證授權的？**

> 兩套並行：後台（React Admin）用 JWT + Role-based Claims，Token 含 WorkerId 和 Role，有效期 4 小時；前台（MVC）用 Cookie Authentication，支援 Google OAuth 第三方登入。兩套機制因為使用場景不同——SPA 適合 JWT，傳統 MVC 更適合 Cookie。

**追問：JWT 放哪裡？有什麼安全考量？**

> JWT 存在 localStorage。更安全的方式是 httpOnly Cookie，可以防 XSS 讀取；但 React SPA 的 CSRF 防護會複雜一些。目前選 localStorage 是因為後台只有公司員工使用，風險可接受；若要上公網的 B2C 系統我會改用 httpOnly Cookie。

**追問：Role 怎麼設計的？**

> API 後台有三個角色：`Admin`、`Supervisor`、`Worker`，用 `[Authorize(Roles = "Admin")]` 保護端點。前台有兩個角色：`User`（一般民眾）和 `Case`（服務個案），功能和頁面完全不同，Case 只能看個案活動和申請物資，User 可以報名民眾活動和購物。

---

**Q5：Google OAuth 怎麼接的？**

> 用 `Microsoft.AspNetCore.Authentication.Google` 套件。流程是：前台 → 重導向 Google 授權頁 → 使用者同意 → Google 回調帶 code → 後端用 code 換 access token → 取得 email → 查 DB 是否有這個 email，有就登入，沒有就自動建立帳號。

**追問：如果 Google 回傳的 email 已經被用來建立一般帳號？**

> 這是個值得注意的帳號衝突問題。目前的處理是「先查 email，有就直接用那個帳號登入」，但沒有區分這個 email 是自行註冊的還是 Google 建立的。完整的處理應該要記錄 `AuthProvider`（local / google），避免帳號劫持。這是我知道但目前 demo 環境沒實作完整的地方。

## 三、資料庫

---

**Q6：你的資料庫是 Code First 還是 DB First？**

> DB First。先在 SQL Server Management Studio 設計好資料表和關聯，再用 `Scaffold-DbContext` 指令反向產生 Entity Class 和 DbContext。因為這個專案 DB 設計先行，後端開發跟著走，DB First 自然一些。

**追問：DB First 的缺點是什麼？**

> 每次資料表有變動，要手動重新 Scaffold 或手動更新 Entity Class，沒有 Migration 記錄歷史變更。Code First 有 Migration 可以追蹤 schema 演進，部署新環境也比較方便。這個 trade-off 我接受，因為資料庫設計相對穩定，不需要頻繁遷移。

**追問：N+1 問題你有遇過嗎？**

> 有注意到。EF Core 預設 Lazy Loading 關閉，所以 `Include()` 要明確寫。例如查活動列表的時候有用 `.Include(a => a.Worker)` 避免 N+1。但有些 DbContext 直接查詢的地方我沒有系統性地 profile 過，這是技術債。

---

**Q7：並發問題你怎麼處理的？**

> 在物資庫存扣減的地方有特別處理。ECPay 收到付款成功回調後，扣減庫存用 Raw SQL + Transaction，避免多個訂單同時成功時 Race Condition 超賣。另外有做冪等保護：`EcpayTransaction.Status == "Success"` 時直接 return，防止 ECPay 重送回調時重複扣庫存。

## 四、金流（ECPay）

---

**Q8：ECPay 的付款流程說一遍？**

> 1. 使用者在前台結帳，MVC 呼叫 `EcpayService.CreatePayment`，用訂單資料建立付款表單（含 CheckMacValue 簽章），產生一個 POST form 導向 ECPay。
> 2. 使用者在 ECPay 頁面輸入信用卡資訊。
> 3. 付款後 ECPay 主動 POST 回調到 `/Purchase/EcpayCallback`（後端 webhook）。
> 4. 後端收到後先驗證 CheckMacValue，驗過才更新訂單狀態和扣庫存，最後回傳 `1|OK` 給 ECPay。
> 5. 同時 ECPay 也會把使用者重導向回前台的 `/Purchase/EcpayReturn`。

**追問：CheckMacValue 是什麼？為什麼要驗兩次？**

> CheckMacValue 是 ECPay 定義的 SHA-256 簽章。計算方法是：把所有參數按 key 排序、URL encode、加上 HashKey 和 HashIV 包頭尾，最後 SHA-256。
>
> 為什麼驗兩次：送出付款請求時，我的後端計算 CheckMacValue 附在 form 裡，這樣 ECPay 能確認請求來自合法商家。收到回調時，我再算一次比對 ECPay 給的值，確認這個回調是真的來自 ECPay，不是有人偽造 POST 來騙我更新訂單狀態。

**追問：你有寫測試嗎？**

> 有，這是我特別重視的。我把 CheckMacValue 計算邏輯抽成 `EcpayCheckMacHelper` 靜態類別，然後用 xUnit 寫了 7 個測試：格式驗證、冪等性、參數排序不影響結果、空值過濾（付款和回調模式不同）、CheckMacValue 自身不被計入、不同金額產生不同 hash。金流是最容易出現 bug 產生財務損失的地方，測試覆蓋特別重要。

## 五、AI 功能

---

**Q9：你整合了哪些 AI 功能？怎麼設計的？**

> 三項：GPT 活動文案優化、DALL-E 3 圖片生成、Whisper 語音轉文字。設計重點是**雙 Provider 架構**：一個 `AI:Provider` 設定可以在 OpenAI Direct API 和 Azure OpenAI 之間切換，不需要改程式碼。

**追問：Provider 切換怎麼實作的？**

> 用 Factory Pattern。`OpenAIClientFactory`（Singleton）在啟動時讀取 `appsettings.json` 的 `AI:Provider`，建立對應的 `OpenAIClient`：OpenAI 直接用 API Key，Azure 用 Endpoint + Key。其他 Service（`AzureOpenAIService`、`WhisperService`）從 factory 取得 client，不關心背後是哪個 provider。

**追問：語音轉文字存在哪？**

> 音檔存本機 `api/uploads/case_audio/`，透過 `IFileStorageService` 介面抽象（Strategy Pattern），切換到 Azure Blob 只需要改設定。Whisper API 收到音檔後回傳文字，文字存進 Case 的備註欄位。

## 六、部署與維運

---

**Q10：你怎麼把系統公開展示的？**

> 用 Cloudflare Tunnel（cloudflared）。在本機啟動三個服務後，執行 `start-demo-smart.bat` 建立 Named Tunnel 連接到固定網域，不需要公網 IP 或伺服器費用。三個固定網域：用戶前台、管理後台、API。

**追問：有什麼安全考量嗎？**

> 有幾個：一是 `appsettings.json` 含 DB 連線和 API Key，全部加進 `.gitignore`，版控只有 `appsettings.example.json`。二是 `UseForwardedHeaders` 明確只信任 loopback proxy（因為 cloudflared 從本機連），防止外部偽造 `X-Forwarded-For`。三是靜態檔案有設 Cache-Control header，避免敏感資源被快取。

## 七、你自己加分的問題

---

**Q11：這個專案開發過程中最有挑戰性的部分是？**

> 兩個。第一個是密碼系統零停機遷移——原本密碼明文存，需要改成 BCrypt 但不能讓現有用戶無法登入，設計了漸進 re-hash 策略，登入時自動升級，全部遷移完再移除向下相容邏輯。第二個是 ECPay 的資料一致性——付款成功和庫存扣減是兩個操作，如果中間出錯，或者 ECPay 重送回調，都可能造成超賣或重複扣款，要用 Transaction + 冪等設計同時解決。

---

**Q12：這個專案如果再來一次你會改什麼？**

> 三個方向。一是從一開始就全面用 Repository Pattern，而不是只有 Case 模組有；二是加 Integration Test 或至少有 Testcontainers 跑真正的 DB，目前單元測試只覆蓋純邏輯，Controller 層沒有測試；三是前台的物資申請設計，原本用卡片式電商 UI，後來考量個案使用者的特性改成列表式，但這個決策應該更早做，而不是設計到一半才想到。

---

**Q13：你對軟體品質的理解是什麼？**

> 品質不只是「功能能跑」，還有：一是可維護性，別人（或三個月後的自己）能快速理解並修改；二是安全性，不讓使用者的資料暴露在風險下；三是可靠性，在邊界情況和錯誤輸入下也能正確處理，而不是靜默失敗。這個專案我做過一次 OWASP 式的資安審查，也補了 Global Exception Handler，就是對這些的具體實踐。
