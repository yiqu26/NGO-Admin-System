## 🎯 系統概覽

**NGO 案件管理系統**是一個完整的非營利組織管理平台，採用**現代化前後端分離架構**，提供個案追蹤、活動管理、物資配送、志工協作等功能。

### 系統定位

- **管理端**：供 NGO 內部員工使用（管理員、督導、社工）
- **用戶端**：供一般民眾和個案使用（活動報名、物資認購）
- **整合端**：統一的後端 API 服務，支援多種前端應用

### 使用者角色

| 角色 | 系統 | 權限等級 |
|------|------|----------|
| **管理員** | React 後台 | 完整系統權限 |
| **督導** | React 後台 | 管理權限（無刪除核心資料） |
| **員工** | React 後台 | 基本操作權限 |
| **一般民眾** | MVC 前台 | 活動報名、物資認購 |
| **個案** | MVC 前台 | 個案專屬活動、物資申請 |

#### ECPay 金流整合（技術亮點）

**付款流程**：
```
1. 用戶選擇物資 → 建立訂單（PaymentStatus: "待付款"）
2. 生成 CheckMacValue（SHA256 加密）
3. 跳轉到 ECPay 付款頁面
4. 用戶完成付款
5. ECPay 回調 ReturnURL（Server-to-Server）
   → 驗證 CheckMacValue
   → 更新訂單狀態為 "已付款"
   → 扣除庫存
6. ECPay 跳轉 ClientBackURL（Browser redirect）
   → 顯示付款成功頁面
```

**安全機制**：
- **CheckMacValue 驗證**：防止偽造回調
- **SHA256 加密**：雙向參數簽章
- **Server 回調**：確保訂單更新可靠性
- **Dual URL 設計**：ReturnURL（後端） + ClientBackURL（前端）

詳細技術分析請參考：[04-ECPay綠界金流技術分析與面試指南.md](04-ECPay綠界金流技術分析與面試指南.md)

### 4. ECPay 金流整合（Payment Processing）

**核心檔案**：
- `Services/EcpayService.cs`（~340 行）
- `Controllers/PurchaseController.cs`（~1170 行）

**關鍵方法**：

1. **CreatePayment()**：生成 ECPay 付款表單
```csharp
public string CreatePayment(UserOrder order, string returnUrl, string clientBackUrl)
{
    // 1. 組合付款參數
    var parameters = new Dictionary<string, string>
    {
        ["MerchantTradeNo"] = order.OrderNumber,
        ["TotalAmount"] = order.TotalAmount.ToString("0"),
        ["ReturnURL"] = returnUrl,          // Server 回調
        ["ClientBackURL"] = clientBackUrl   // 瀏覽器跳轉
    };

    // 2. 生成 CheckMacValue（SHA256 加密）
    var checkMacValue = GenerateCheckMacValue(parameters, _hashKey, _hashIv);
    parameters["CheckMacValue"] = checkMacValue;

    // 3. 產生 HTML Form
    return GenerateHtmlForm(parameters);
}
```

2. **ProcessCallback()**：處理 ECPay 回調
```csharp
public async Task<bool> ProcessCallback(IFormCollection form)
{
    // 1. 驗證 CheckMacValue
    var receivedCheckMac = form["CheckMacValue"];
    var calculatedCheckMac = GenerateCallbackCheckMacValue(form, _hashKey, _hashIv);

    if (receivedCheckMac != calculatedCheckMac)
        return false; // 拒絕偽造請求

    // 2. 更新訂單狀態
    var merchantTradeNo = form["MerchantTradeNo"];
    var rtnCode = form["RtnCode"];

    if (rtnCode == "1") // 付款成功
    {
        order.PaymentStatus = "已付款";
        await UpdateInventoryAndEmergencyNeeds(order);
    }

    return true;
}
```

**安全機制**：
- ✅ SHA256 加密驗證
- ✅ Dual URL 模式（ReturnURL + ClientBackURL）
- ✅ Server-to-Server 回調確保資料可靠性
- ✅ CheckMacValue 防止參數篡改

### 5. ECPay 金流安全機制（面試重點）

**雙重驗證**：
```
【發送付款請求】
1. 組合參數（MerchantTradeNo, Amount, ReturnURL...）
2. 產生 CheckMacValue = SHA256(參數 + HashKey + HashIv)
3. 發送到 ECPay

【接收付款回調】
1. ECPay 回傳參數 + CheckMacValue
2. Server 重新計算 CheckMacValue
3. 比對是否一致 → 驗證通過 → 更新訂單
4. 若不一致 → 拒絕請求（防止偽造）
```

**Dual URL 設計**：
- **ReturnURL**（Server-to-Server）：確保訂單更新可靠
- **ClientBackURL**（Browser redirect）：提供用戶體驗

**面試說明範例**：
> "我們的 ECPay 整合採用雙重安全機制：首先使用 SHA256 加密產生 CheckMacValue 防止參數篡改；其次採用 Dual URL 模式，ReturnURL 確保 Server 端可靠接收付款結果並更新訂單，ClientBackURL 負責將用戶導向成功頁面。這種設計即使 Client 端跳轉失敗，Server 端也能正確更新訂單狀態，確保資料一致性。"

### Q2: ECPay 金流如何確保安全性？

**答**：
1. **CheckMacValue 驗證**：使用 SHA256 加密，防止參數篡改
2. **Dual URL 設計**：
   - ReturnURL（Server-to-Server）：確保訂單可靠更新
   - ClientBackURL（瀏覽器跳轉）：提供用戶回饋
3. **雙向驗證**：發送請求時加密，接收回調時驗證
4. **狀態檢查**：只有 RtnCode = "1" 才更新訂單為已付款

### Q3: 如何處理不同角色的權限控制？

**答**：
我們實作了基於角色的權限控制（RBAC）：
1. **後端**：使用 ASP.NET Core `[Authorize]` + Role Claims
2. **前端**：使用 `useUserRole()` hook 控制 UI 顯示
3. **資料隔離**：員工只能查看被指派的個案
4. **操作限制**：督導可管理但不能刪除核心資料
